`invoke_command:/spec` 对本项目进行一次与"企业级生产就绪审计（Prompt A）"完全垂直的深度自检审计。 
 另一个审计智能体正在并行审查：系统设计、可观测性三支柱、弹性工程、CI/CD、 
 测试策略、API设计、安全认证、云原生容器化、数据库工程和工程文化。 
 
 本轮聚焦以上维度完全未覆盖的 8 个正交维度，且将"优化、简化、强化、加固、 
 安全审计"贯穿每个维度的每一项检查——不只找出问题，而是： 
 
 【铁律：证据优先，禁止猜测】 
 在报告任何发现之前，你必须完整读取所有相关代码文件。 
 不允许基于猜测或假设得出结论。每个发现必须引用具体文件名和行号。 
 
 【核心审计四问】 
 对每一项检查，必须同时回答四个问题： 
 ① 优化：这里有性能或资源浪费吗？ 
 ② 简化：这里有不必要的复杂度吗？（切斯特顿围栏：改之前先理解为何存在） 
 ③ 强化：这里有脆弱性或容错缺失吗？ 
 ④ 安全：这里有可被利用的漏洞吗？ 
 
 【审计的根本目标】 
 以"企业级工程师的成长学习"为核心，不只找出问题： 
 1. 对每个发现，解释其背后的企业工程原理和如果忽视会付出的商业代价 
 2. 指出该能力在就业市场中的价值定位 
 3. 按企业标准实施改造，以 ADR 格式记录每个关键决策 
 即使某改造对当前规模"超前"，只要是行业标准实践，就应实施并注明原因。 
 
 ━━━ 维度 1：领域建模与业务逻辑质量（Domain Modeling）━━━ 
 
 个人项目与企业级代码最深层的差距在于：企业级代码以"业务语言" 
 组织，而非"技术概念"组织。 
 
 【战略设计检查】 
 - 识别系统中的"核心子域"（Core）、"支撑子域"（Supporting） 
   和"通用子域"（Generic）。当前代码是否按此分层投入不同质量的设计精力？ 
 - 是否存在清晰的"领域边界"（Bounded Context）？还是所有逻辑混杂在 
   一个扁平层中？若存在多个业务概念，它们之间的上下文映射是什么关系？ 
 
 【战术设计检查】 
 - 是否使用了充血模型（Rich Domain Model）而非贫血模型（Anemic Domain Model）？ 
   · 业务规则是在领域对象内部强制执行，还是散落在 service/handler 层？ 
   · 是否有 Value Object 封装不变性（如 Email、Money、ISBN）？ 
   · 是否有 Aggregate 边界保证业务不变量的强一致性？ 
   · 是否有 Domain Event 表达"业务上发生了什么"？ 
 - 是否存在"贫血模型反模式"：只有 getter/setter 的结构体，业务逻辑 
   全在外部服务中，导致任何业务规则变更都需要修改多个文件？ 
 - 是否有 Ubiquitous Language（统一语言）？代码中命名是否和业务术语一致？ 
   还是充斥技术术语（"process"/"handle"/"data"/"info"/"util"）？ 
 
 【架构分层检查（含简化审计）】 
 - 依赖方向是否正确：Domain → Application → Infrastructure → Presentation？ 
   内层是否对外层有依赖（违反依赖倒置原则）？ 
 - 是否存在"框架污染领域层"（如 ORM tag 直接出现在领域对象中）？ 
 - 是否可以在不修改业务逻辑的情况下替换数据库？替换 HTTP 框架？ 
 - 是否考虑了 CQRS（命令查询职责分离）？ 
   写路径（Command）和读路径（Query）是否有不同的性能和一致性需求？ 
   当前混合模型在什么规模下会成为瓶颈？ 
 
 【简化专项——领域层】 
 - 找出所有"业务逻辑泄漏"到 handler/controller 层的位置，逐一列出 
   并给出内移到领域层的具体重构方案。 
 - 找出所有命名不能自解释其业务含义的函数/结构体，给出改名建议。 
 - 找出所有跨越多个层次的"上帝函数"（做超过一件事），给出拆分方案。 
 
 ━━━ 维度 2：性能工程（Performance Engineering）━━━ 
 
 注意：这不是"加索引和加缓存"——而是系统性、可重复的性能科学， 
 是 SRE 的核心职责。 
 
 【性能基线建立】 
 - 是否有 go test -bench 基准测试？哪些关键路径有性能基准？ 
   无基准 = 无法感知性能回退（regression）。 
 - 是否有 CPU/内存 profiling 数据（pprof）？ 
   · 是否跑过 CPU 火焰图？找到热点函数了吗？ 
   · 是否跑过内存分配分析？GC 压力如何（allocs/op）？ 
   · 是否跑过 goroutine 泄漏分析（goleak）？ 
 - 对以下典型场景建立性能基线并给出当前实测数据： 
   单请求延迟（P50/P95/P99）、吞吐量峰值（RPS）、内存稳态占用。 
 
 【负载测试与容量规划】 
 - 是否有负载测试脚本（k6/vegeta/hey）？ 
   · 并发 100/1000/10000 请求时的行为如何？ 
   · 延迟开始非线性增长的拐点在哪里？ 
   · 服务开始返回错误的并发量是多少？ 
 - 是否做过容量规划？ 
   · 当前单实例能支撑多少 DAU？ 
   · 什么时候需要读写分离/水平扩展？给出量化估算。 
 - 识别系统理论瓶颈（按 USL 通用扩展定律）： 
   数据库连接池 → 单一热点表 → 内存限制 → CPU 限制，给出当前余量。 
 
 【代码层优化专项（必须给出所有命中位置的文件名+行号）】 
 - N+1 查询：找出所有循环内的数据库查询或未用 JOIN 的关联查询， 
   给出具体修复方案（批量查询/JOIN/预加载）。 
 - 序列化热路径：JSON 编解码是否在高频路径上？是否有必要使用 
   更高效方案（sonic/easyjson）？ 
 - 内存分配：频繁小对象分配是否可用 sync.Pool 复用？ 
 - 字符串操作：循环中的字符串拼接是否用了 strings.Builder？ 
 - 值传递 vs 指针：大结构体是否错误地按值传递？ 
 - 并发优化：是否有本可并发但实际串行的独立 IO 操作（可用 errgroup 并发化）？ 
 - 预编译：正则表达式、SQL 语句是否在全局/初始化时预编译？ 
 
 ━━━ 维度 3：SRE 实践与可靠性工程（Site Reliability Engineering）━━━ 
 
 注意：这与 Prompt A 的"弹性工程"不重叠——弹性工程关注代码层面的模式， 
 SRE 关注如何用工程方法管理可靠性目标和故障响应。 
 
 【SLI/SLO/SLA 定义】 
 - 是否为核心用户旅程定义了 SLI（服务水平指标）？ 
   · 可用性 SLI：成功请求率（非 5xx） 
   · 延迟 SLI：P99 延迟低于阈值的请求比例 
   · 正确性 SLI：返回正确结果的请求比例 
 - 是否基于 SLI 定义了 SLO（服务水平目标）？ 
   · 如"滚动30天内，99.5% 的请求在 200ms 内返回 2xx" 
   · SLO 阈值是否经过实测数据验证（而非拍脑袋定的99.99%）？ 
 - 是否有 Error Budget（错误预算）的概念？ 
   · 当前滚动窗口内消耗了多少错误预算？ 
   · 是否有根据错误预算余量决定"发布新功能"还是"投入稳定性工作"的机制？ 
 - 是否有针对每个 SLO 的 Burn Rate Alert（烧毁率告警）？ 
   企业标准：同时使用"快速烧光"（高烧毁率，短窗口）和 
   "慢速泄漏"（低烧毁率，长窗口）两类告警。 
 
 【混沌工程（Chaos Engineering）】 
 - 是否进行过任何形式的故障注入实验？ 
 - 设计至少 3 个对本项目最相关的混沌实验： 
   · 实验一：数据库连接全部断开时，系统如何响应？健康检查能否正确反映？ 
   · 实验二：外部服务响应延迟增加至 5s 时，是否引发级联故障？ 
   · 实验三：高并发下重启服务，是否丢失正在处理的请求？ 
   对每个实验：定义稳态假设、实验方法、成功标准、当前系统是否能通过。 
 - 推荐工具：toxiproxy（网络故障注入）、gofail（Go 层面故障注入）。 
 
 【Superpowers 风格的系统性故障分析（Defense-in-Depth）】 
 - 对每个已知的失败模式，执行"纵深防御"检查： 
   · 在哪一层最先发现这个故障？ 
   · 如果该层的检测失败了，下一层能兜住吗？ 
   · 目前防御的层次：输入验证层 / 业务逻辑层 / 数据库约束层 / 监控告警层 
     是否每层都有独立的防御机制，而非只依赖单点防御？ 
 - 找出所有"假设外部输入是干净的"的代码位置（即缺少防御性验证的位置）， 
   逐一列出并给出在该层添加防御的方案。 
 
 【事故响应与 Postmortem 文化】 
 - 是否有 docs/runbook.md 包含最可能出现的 5 类故障的排查步骤？ 
   每个条目：症状 → 可能原因 → 排查命令 → 缓解步骤 → 根治方案 
 - 是否有 Postmortem 模板（docs/postmortem-template.md）？ 
   企业标准：无责 Postmortem，包含： 
   · 事件时间线、影响范围（受影响用户数、持续时间、错误预算消耗） 
   · 根本原因分析（5个为什么） 
   · 行动项（含负责人和 DDL） 
 
 ━━━ 维度 4：代码质量可量化指标（Code Quality Metrics）━━━ 
 
 "代码质量"不应只是主观感受。企业工程师用数据说话。 
 Agent Skills 的 /code-simplify 哲学：目标不是更少的行数， 
 而是更容易阅读、理解、修改和调试的代码。 
 
 【复杂度与可维护性指标（必须给出实测数值）】 
 - 使用静态分析工具评估并给出每项实测值： 
   · 圈复杂度（Cyclomatic Complexity）：每个函数的分支复杂度 
     企业基准：单函数 ≤ 10，> 15 是重构红线，> 20 是严重警告 
   · 认知复杂度（Cognitive Complexity）：代码难以理解的程度 
   · 函数长度：超过 50 行的函数完整列表 
   · 包耦合度：哪些包之间有过度耦合？ 
   · 内聚度：每个包的职责是否单一明确？ 
 - 运行 golangci-lint 完整规则集，给出所有 linter 的报告结果。 
   至少启用：gocyclo, gocritic, gosimple, ineffassign, staticcheck, 
   unparam, unused, revive, godot, goconst, gomnd, nestif 
 - 使用 deadcode 工具扫描，列出所有未使用的导出函数（死代码）。 
 - 使用 dupl 工具扫描重复代码块，并给出抽象建议。 
 
 【Agent Skills 五轴代码审查（Five-Axis Review）】 
 对每个核心文件，按以下五轴逐一评估： 
 ① 正确性（Correctness）：代码是否在所有路径上都做了正确的事？ 
   边界情况是否被处理？并发行为是否安全？ 
 ② 可读性（Readability）：代码是否不需要注释就能被理解？ 
   命名是否传达了意图？控制流是否简单清晰？ 
 ③ 可维护性（Maintainability）：修改一处需要改多少地方？ 
   是否容易安全地扩展？是否有清晰的关注点分离？ 
 ④ 性能（Performance）：是否存在不必要的计算、内存分配或 IO 操作？ 
 ⑤ 安全性（Security）：是否存在输入验证缺失、权限检查遗漏或数据泄露风险？ 
 每轴使用：🟢 通过 / 🟡 改进空间 / 🔴 必须修复 
 
 【技术债务量化】 
 - 按以下框架量化当前技术债： 
   · 主债（Principal）：需要多少工时修复所有已知问题？分维度估算。 
   · 利息（Interest）：每次功能迭代因技术债额外花费的时间（减速因子）。 
   · 债务比率：技术债工时占总开发工时的估算比例。 
     企业参考：> 25% 是警告线，> 40% 是危机线。 
 - 列出"技术债热点地图"：高改动率 × 高复杂度 = 最高风险区域。 
 - 魔法数字/字符串（magic numbers/strings）：扫描并提取为具名常量。 
 - 错误处理模式一致性：是否混用了多种错误处理风格？统一建议。 
 
 【代码简化专项——Chesterton 围栏协议】 
 对每一项简化建议，必须先回答： 
 "这段复杂代码存在的原因是什么？" 
 如果无法找到合理原因 → 列为"安全简化" 
 如果原因可疑但不确定 → 列为"需确认后简化" 
 如果原因合理 → 列为"保留并添加注释说明原因" 
 
 ━━━ 维度 5：供应链安全与合规治理（Supply Chain & Compliance）━━━ 
 
 这是 2025-2026 年最热门的企业安全话题，与 Prompt A 的"依赖漏洞扫描" 
 完全不同——本维度关注整个软件生命周期的供应链完整性治理。 
 
 【SBOM（软件物料清单）生成】 
 - 是否为项目生成了 SBOM（Software Bill of Materials）？ 
   · 使用 syft 生成 CycloneDX 或 SPDX 格式的 SBOM。 
   · SBOM 应包含：所有直接和传递依赖、版本、license 信息、已知 CVE 关联。 
   · SBOM 生成是否集成到构建流程（每次构建自动更新）？ 
 - 是否对构建产物（Docker 镜像）进行了签名（Artifact Signing）？ 
   · 使用 cosign（Sigstore 项目）对容器镜像签名，确保部署的镜像来自可信构建。 
   · 这是2026年企业供应链安全基础要求（EU Cyber Resilience Act 推动）。 
 - 是否有 SLSA 等级评估？ 
   · Level 1：构建脚本可用，有构建日志 
   · Level 2：有版本控制，构建服务可信 
   · Level 3：构建流程不可篡改，有出处证明（Provenance） 
   当前项目在哪个等级？如何提升？ 
 
 【许可证（License）合规】 
 - 使用 go-licenses 工具生成 license 报告。 
   · GPL/LGPL 类（Copyleft）是否被引入？对商业化的影响是什么？ 
   · 企业项目通常要求全面使用 MIT/Apache 2.0/BSD 等宽松许可证。 
 - 是否有 LICENSE 文件和 NOTICE 文件？ 
 
 【数据隐私合规框架】 
 - 是否有数据分类文档？哪些字段属于 PII / 敏感个人信息 / 非个人数据？ 
 - 是否实现了数据生命周期管理？ 
   · 数据保留策略：不同类型数据的最长存储期限？ 
   · 数据删除机制：是否支持"被遗忘权"（GDPR Article 17）？ 
   · 数据最小化：每个接口是否只收集完成功能所必需的最少数据？ 
 - GDPR/CCPA/PIPL 各自的核心义务映射到代码层面是什么实现？ 
 
 ━━━ 维度 6：异步架构与事件驱动设计（Async & Event-Driven Architecture）━━━ 
 
 这是 Prompt A 完全未涉及的架构维度，是微服务和高可扩展系统的核心模式。 
 
 【当前同步边界识别（结合安全审计）】 
 - 识别所有"目前是同步但应该是异步"的操作： 
   · 哪些操作让用户等待，但实际上不需要立即完成？ 
   · 哪些操作在失败时需要重试，但当前没有持久化重试机制（崩溃即丢失）？ 
   · 哪些长时间运行的操作没有超时保护，可能被攻击者用于耗尽资源（DoS）？ 
 - 对每个识别出的操作，给出是否需要异步化的判断依据和方案。 
 
 【消息队列与事件总线设计】 
 - 如果当前没有消息队列，设计适合本项目的最小化方案： 
   · 单机部署：PostgreSQL LISTEN/NOTIFY 实现轻量事件总线？ 
     解释 Outbox Pattern（发件箱模式）比直接调用消息队列更可靠的场景。 
   · 分布式部署：NATS / RabbitMQ / Kafka 的选型依据对比。 
 - 是否需要 Transactional Outbox（事务性发件箱）？ 
   · 典型问题：数据库写入成功但消息发送失败导致数据不一致。 
   · 在本项目中哪些写操作需要此保障？ 
 
 【事件溯源与最终一致性（含安全审计）】 
 - 是否有需要完整审计轨迹的业务操作？ 
   · 事件溯源（Event Sourcing）是否是更适合的方案？ 
   · 解释 Event Sourcing 与 CRUD 的根本差异、优势和代价。 
 - 是否存在跨实体的操作需要考虑最终一致性？ 
   · Saga 模式（编排式 vs 编舞式）各适合什么场景？ 
   · 当前代码中是否有"部分成功、部分失败"的操作没有补偿机制？ 
 - 安全维度： 
   · 事件流（event log）是否防篡改？是否有消息认证码（HMAC）保护？ 
   · 消费者是否对重复消息（重试导致）做了幂等性保护？

   · 是否有 testhelpers/fixtures/ 包集中管理 mock/stub/fake？ 
   · 是否使用 gomock/testify/mock 生成接口 mock？ 
   · 测试数据是否使用 Factory 模式（而非硬编码值）构建？ 
   · 每个修复是否都有对应的回归测试证明它被修复了（而非只是症状消失）？ 
 - 验证完成（Verification-Before-Completion）： 
   · 当前代码中是否有任何"只运行 happy path"的测试？ 
   · 是否有针对每个错误处理分支的单元测试？ 
   · 是否有任何"TODO: add test"注释？（反模式，立即清零） 
 
 【工具链标准化（含代码简化工具集成）】

   至少包含：make test, make lint, make build, make run, 
   make migrate, make seed, make clean, make generate, 
   make bench（性能基准）, make audit（安全扫描）, make simplify（运行简化检查） 
 - 是否有 tools.go 锁定所有开发工具版本？ 
 - 代码简化工具链是否集成： 
   · gofmt / goimports（格式化） 
   · golangci-lint（综合 lint，含圈复杂度） 
   · gocritic（Go 惯用法检查） 
   · deadcode（死代码） 
   · dupl（重复代码检测） 
 
 ━━━ 维度 8：深度安全审计（Security Hardening & Attack Surface Reduction）━━━ 
 
 注意：这与 Prompt A 的"安全深度"维度完全不同——Prompt A 关注认证授权设计 
 和威胁建模框架，本维度关注代码层面的具体漏洞和攻击面收缩。 
 
 【OWASP Top 10 逐条对照审计（必须给出每条的具体代码位置）】 
 参照 OWASP Top 10（2021版）和 OWASP Top 10 for APIs（2023版）， 
 对每一条给出：是否存在 / 具体位置（文件名+行号）/ 修复方案 / 修复优先级 
 
 A01 失效的访问控制（Broken Access Control）： 
 · 是否有水平越权风险（用户A访问用户B的数据）？ 
 · 是否有垂直越权风险（普通用户执行管理员操作）？ 
 · 目录遍历、路径穿越漏洞是否已防护？ 
 
 A02 加密失败（Cryptographic Failures）： 
 · 是否有明文存储/传输敏感数据（密码、令牌、PII）？ 
 · 是否使用了已废弃的哈希算法（MD5/SHA1 用于密码）？ 
 · TLS 配置是否强制最低版本（TLS 1.2+）？ 
 
 A03 注入（Injection）： 
 · SQL 注入：是否有字符串拼接 SQL（即使已修复也要验证彻底性）？ 
   使用 sqlc / ent 或参数化查询是否全面覆盖？ 
 · 命令注入：是否有 exec.Command 使用用户输入？ 
 · LDAP / XPath / NoSQL 注入的可能性？ 
 · 模板注入：如果有模板引擎，是否使用了 html/template 而非 text/template？ 
 
 A04 不安全设计（Insecure Design）： 
 · 是否有限速（Rate Limiting）防止暴力破解？ 
 · 是否有业务逻辑漏洞（如负数金额、跳过必要步骤）？ 
 · 敏感操作是否有多因素确认机制？ 
 
 A05 安全配置错误（Security Misconfiguration）： 
 · 默认凭据是否全部变更？ 
 · 错误信息是否暴露了系统内部路径/栈追踪？ 
 · 调试模式 / 开发端点是否在生产环境中关闭？ 
 · 不必要的功能/端点/方法是否已禁用？ 
 
 A06 易受攻击和过时的组件（Vulnerable Components）： 
 · 运行 govulncheck，列出所有存在已知 CVE 的依赖及其修复版本。 
 · Docker 基础镜像是否使用了有已知漏洞的版本？ 
 · 运行 trivy 或 grype 对镜像进行全面漏洞扫描。 
 
 A07 身份识别和认证失败（Authentication Failures）： 
 · 会话 token 是否有足够的熵（随机性）？ 
 · 失败登录是否有计数和锁定机制？ 
 · 密钥是否安全存储（非代码硬编码，使用环境变量或 secret manager）？ 
 · JWT 是否验证了算法（防止 alg=none 攻击）？签名密钥是否足够强？ 
 
 A08 软件和数据完整性失败（Integrity Failures）： 
 · 是否验证了所有外部数据源（第三方 webhook、文件上传）的完整性？ 
 · 反序列化操作是否对输入数据类型进行了严格验证？ 
 · 依赖包是否通过校验和验证（go.sum 是否提交到版本控制）？ 
 
 A09 安全日志记录和监控失败（Security Monitoring Failures）： 
 · 是否记录了所有认证事件（成功/失败/异常）？ 
 · 是否记录了权限提升事件？ 
 · 日志是否防篡改（写入后不可修改）？ 
 · 是否有异常行为检测（如同一用户短时间内大量请求）的监控？ 
 
 A10 服务器端请求伪造（SSRF）： 
 · 是否有对用户提供的 URL 发起请求的功能？是否有 SSRF 防护？ 
 · 元数据服务地址（169.254.169.254）是否在黑名单中？ 
 · 对内网 IP 段的请求是否被阻断？ 
 
 【三层安全边界系统（Three-Tier Boundary System）】 
 参照 Agent Skills security-and-hardening 的三层边界模型： 
 
 第一层：输入边界（Entry Boundary） 
 · 所有外部输入是否都经过验证（类型/长度/范围/格式/字符集）？ 
 · 是否有中央化的输入验证层，而非散落在各处？ 
 · 跨站脚本（XSS）：如果有任何 HTML 输出，是否使用了安全模板引擎？ 
 
 第二层：处理边界（Processing Boundary） 
 · 是否遵循最小权限原则（每个组件只有完成工作所需的最小权限）？ 
 · 数据库用户是否只有必要的 CRUD 权限（而非 superuser）？ 
 · 是否有 context deadline 防止长时间运行的操作耗尽资源？ 
 · 文件操作是否有路径规范化（filepath.Clean）防止目录穿越？ 
 
 第三层：输出边界（Output Boundary） 
 · API 响应是否过滤了内部实现细节（DB 错误、堆栈追踪、内部路径）？ 
 · 是否有响应体大小限制（防止巨型响应耗尽客户端/代理内存）？ 
 · 是否在所有适合的端点设置了安全 HTTP 响应头？ 
   （HSTS、X-Frame-Options、CSP、X-Content-Type-Options、Referrer-Policy） 
 
 【供应链安全（代码层面的信任链）】 
 - go.sum 是否被提交到版本控制（确保依赖哈希一致性）？ 
 - 是否有 Dependabot 或 Renovate 自动扫描依赖更新？ 
 - Docker 基础镜像是否 pin 到 digest（而非 tag）防止供应链攻击？ 
 - 是否有构建重现性（Reproducible Builds）的考量？ 
 
 【密钥管理深度审计】 
 - 在代码库（含历史提交）中全面扫描可能的密钥泄漏： 
   · 使用 detect-secrets 或 truffleHog 扫描 git 历史。 
   · 环境变量是否都通过 os.Getenv 而非硬编码读取？ 
   · 日志中是否有可能打印到密钥的位置？ 
   · 测试文件中是否有真实密钥（而非 mock 值）？ 
 - 密钥轮换：是否支持密钥轮换而不需要重新部署？ 
 - 密钥最小暴露原则：密钥是否只在需要的范围内可见？
