# ADR-023: 数据隐私分类与删除权实现

> **企业理由**：GDPR/PIPL/CCPA 要求企业明确持有的个人数据、保留期与删除机制。缺失删除能力是直接的合规违规（GDPR 罚款上限为全球营收 4%），且 SOC 2/ISO 27001 审计必查项。

## Context

审计（维度5 / 5.8）发现：

- 无数据分类文档；无 PII 字段台账。
- `userService` 仅有 `createUser/verifyUser/getUserById`，**无删除/停用 API**。
- 无数据保留/删除策略，无"被遗忘权"实现。

虽然当前仅存储 `username` + `password_hash`（PII 极少），但缺失删除能力在引入任何用户增长时即成合规阻塞。

## Decision

1. **数据分类文档**：新增 `docs/data-classification.md`，定义分类、PII 台账、保留期、法规→代码映射。
2. **删除机制三档**：
   - `deactivateUser`：软删除（可逆），撤回访问，保留记录。
   - `anonymizeUser`（**默认推荐**）：GDPR Art.17 假名化——抹除 username/password_hash，保留占位标识以维持审计/外键完整性与聚合统计。
   - `deleteUser`：硬删除，用于无留存义务的彻底删除。
3. **数据最小化**：维持仅采集 username + hash，不引入 email/电话等非必需 PII。

## Consequences

- 优势：具备可演示的合规能力（DSAR 删除）；审计有据；引用完整性不被破坏。
- 劣势：匿名化而非硬删除会保留占位行——这是引用完整性与"彻底删除"的标准权衡，已在文档说明。
- 后续：补齐审计日志保留期自动清理作业与 DSAR 受理流程（治理层，非纯代码）。
- 工程权衡：默认匿名化（而非硬删除）优先保护系统完整性，符合多数企业实践；硬删除作为显式选项保留。
