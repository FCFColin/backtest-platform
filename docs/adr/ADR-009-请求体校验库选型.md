# ADR-009: 请求体校验库选型（zod vs joi vs class-validator）

> **企业理由**：运行时类型校验是防止畸形请求绕过编译时检查、引发业务异常或安全漏洞的最后一道防线，统一的校验库可降低维护成本并保证请求入口安全。

## Context
Express路由中14处使用`req.body as Type`类型断言，TypeScript类型仅在编译时检查，运行时req.body可包含任意数据。攻击者可发送畸形请求体绕过编译时类型约束，导致业务逻辑处理意外数据，引发运行时异常或安全漏洞。需要运行时校验库在请求入口处拦截非法数据。

候选方案：
- **zod**：TypeScript-first schema校验库，支持类型推导
- **joi**：Node.js生态老牌校验库，Hapi生态原生支持
- **class-validator**：基于装饰器的校验库，与TypeScript类配合使用

## Decision
采用zod作为运行时请求体校验库。

## Consequences
- (+) TypeScript-first，`z.infer<typeof schema>`自动推导类型，无需维护两套类型定义（一套TypeScript接口、一套校验schema）
- (+) 社区活跃度最高（GitHub 30k+ stars），生态完善，与Express中间件（如zod-express-middleware）集成成熟
- (+) 支持复杂场景（联合类型、条件校验、变换/transform），满足回测参数的多样化校验需求
- (+) 零运行时依赖，bundle size小，适合Serverless/Edge场景
- (-) 放弃class-validator的装饰器风格——装饰器与Express函数式路由不够契合，需要额外class-transformer配合plainToInstance
- (-) 放弃joi的成熟度——joi类型推导不如zod，需要手动维护TypeScript类型与joi schema的对应关系
- (-) zod v4（若升级）可能带来breaking change，需关注版本策略
