# ADR-034: 服务端租户持久化 + 前端认证会话

> **企业理由**：组合、命名配置与回测历史此前仅存于浏览器 `localStorage`，跨设备不可用、无法协作、清缓存即丢失，也无法做配额/计费。多租户 SaaS 必须把这些数据落到服务端并按租户隔离；前端需要真正的登录会话与组织切换能力。

## Decision（决策内容）

### 后端

- 新增基于 `withTenant`（RLS）的仓储：`portfolioRepo`、`savedConfigRepo`、`backtestRunRepo`，落到 `portfolios` / `saved_configs` / `backtest_runs`（JSONB 存灵活载荷）。
- 新增 CRUD 路由 `/api/v1/portfolios`、`/api/v1/configs`、`/api/v1/runs`（`jwtAuth + resolveTenant + requireTenant + requirePermission`）。
- `adminRoutes` 的 `backtest_history` 改为租户作用域查询。

### 前端

- `src/utils/authTokens.ts`：access token 存内存、refresh token 存 `localStorage`，带并发去重的刷新逻辑（避免与 `apiClient` 循环依赖）。
- `src/utils/apiClient.ts`：注入 `Authorization: Bearer`，401 自动刷新并重试一次，兼容 `x-api-key`。
- `src/store/authStore.ts`（Zustand）：管理 `user/org/orgs`，提供 `loginPassword/register/acceptInvite/switchOrg/loadOrgs/logout/init`。
- 新增页面：`LoginPage`、`AccountPage` 接入真实数据、`Navbar` 内组织切换器与用户菜单。
- `BacktestPage` 命名配置改走 `/configs` API，未登录回退 `localStorage`，并提供一次性本地配置迁移。

## Consequences（后果）

### 正面

- 数据跨设备可用、可协作、可计费；隔离由 RLS 强制。
- 前端具备完整登录/刷新/组织切换体验。

### 负面

- 引入前后端会话状态复杂度（token 刷新、组织切换需重签）。
- 未登录用户仍依赖 `localStorage`，需维护双路径直至强制登录。
