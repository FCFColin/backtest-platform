# ADR-008: 语言精简（4 语言 → Go + TypeScript/React）

> **企业理由**：多语言架构虽然让每种语言在最擅长的领域发挥优势，但带来了显著的运维复杂度、团队门槛和调试难度。对于中小团队，2 语言架构（Go 用于后端服务、TypeScript 用于前端+API 网关）在性能、生态和开发效率之间取得最佳平衡。

| ------ | --------------------- |
| 状态   | 已接受                |
| 日期   | 2026-06-23            |
| 决策者 | 架构组                |
| 范围   | 系统级                |
| 取代   | ADR-001（多语言架构） |

## Context（背景和驱动力）

ADR-001 于 2025-01-15 决定采用四语言微服务架构（Rust + Node + Go + Python），各语言职责明确。该决策在当时合理，但随着项目演进，多语言的负面后果日益显著：

### Rust 引擎的问题

1. **编译时间长**：Rust release 编译（LTO + opt-level=3）在 CI 中耗时 5-10 分钟，是 Node/Go 构建时间的 10 倍以上，严重影响 CI 效率。
2. **团队门槛高**：Rust 的所有权系统、生命周期、零成本抽象理念对大多数开发者有较高学习曲线，招聘和培训成本显著高于 Go。
3. **生态局限**：Rust 的 Web 框架生态（Actix-Web/Axum）相比 Go 的 Gin/Echo 在中间件、文档、社区支持方面差距明显。
4. **调试困难**：跨服务调用链追踪在 Rust 边界断裂，Rust 的 OTel 集成不如 Go 成熟。
5. **性能过剩**：回测平台的计算瓶颈在 I/O（数据获取）而非 CPU。Go 的并发模型（goroutine + channel）足以覆盖当前回测计算需求，Go 的数值计算库（gonum）可满足矩阵运算需求。

### Python 脚本的问题

1. **子进程开销**：Node.js 通过 `spawn` 调用 Python，每次启动 Python 解释器约 200ms，高并发下信号量（=3）成为瓶颈。
2. **双运行时**：需要同时维护 Node.js 和 Python 依赖，Docker 镜像需包含 Python 运行时。
3. **已有 Go 替代**：baostock 已有纯 Go 实现（`data-fetcher/baostock/`），akshare/yfinance 的数据可通过 Go HTTP 客户端直接调用其 REST API 或使用 Go 原生库替代。
4. **类型安全缺失**：Python 的动态类型在数据管道中容易引入运行时错误，Go 的静态类型可在编译期捕获。

### Go 的优势（作为统一后端语言）

1. **并发模型**：goroutine + channel 天然适合 I/O 密集型数据服务和 CPU 密集型计算，无需 async/await 或 Promise 链。
2. **编译速度**：Go 编译速度比 Rust 快 10-50 倍，CI 效率显著提升。
3. **部署简单**：Go 静态链接生成单一二进制，Docker 镜像可使用 scratch/alpine，镜像体积小。
4. **生态成熟**：Gin（Web）、pgx（数据库）、gonum（数值计算）、prometheus（监控）、otel（追踪）均为生产级库。
5. **团队友好**：Go 语法简洁，学习曲线平缓，招聘池大。

### TypeScript/React 的保留理由

1. **前端生态**：React + Vite + Tailwind 是当前最主流的前端技术栈，无替代必要。
2. **API 网关**：Express 生态成熟，中间件丰富（helmet/cors/rate-limit），TypeScript 提供类型安全。
3. **全栈类型共享**：前后端共享 TypeScript 类型定义，减少接口对接错误。

## Decision（决策）

**将系统从 4 语言精简为 2 语言：Go（后端服务）+ TypeScript/React（前端 + API 网关）。**

### 具体迁移方案

#### Rust 引擎 → Go 引擎

| 原 Rust 模块          | Go 替代方案 | 关键库                                        |
| --------------------- | ----------- | --------------------------------------------- |
| engine.rs（回测核心） | Go 回测引擎 | gonum/mat（矩阵运算）、gonum/stat（统计指标） |
| monte_carlo.rs        | Go 蒙特卡洛 | gonum/stat/dist + sync.Pool 并行              |
| optimizer.rs          | Go 优化器   | gonum/optimize（优化算法）                    |
| analysis.rs           | Go 分析模块 | gonum/stat（相关性/波动率）                   |
| actix-web HTTP        | Gin HTTP    | gin-gonic/gin（已有）                         |

**性能预期**：

- Go 的数值计算性能约为 Rust 的 70-90%（GC 暂停在计算密集场景可控，可通过对象池优化）
- Go 的 HTTP 吞吐量与 Rust 相当（Gin 与 Actix-Web 在中等负载下差异 < 10%）
- 对于蒙特卡洛等 CPU 密集型任务，Go 的 goroutine 并行比 Rust 的 rayon 更易使用

**权衡**：

- 放弃 Rust 的零成本抽象和内存安全保证，换取开发效率和团队可及性
- Go 的 GC 在极端低延迟场景（< 1ms）不如 Rust，但回测平台对延迟要求为秒级，可接受

#### Python 脚本 → Go 数据获取

| 原 Python 模块       | Go 替代方案                                                        |
| -------------------- | ------------------------------------------------------------------ |
| akshare（A股数据）   | Go HTTP 客户端直接调用 akshare REST API，或使用东方财富/同花顺 API |
| yfinance（美股数据） | Go HTTP 客户端调用 Yahoo Finance API（已有纯 HTTP 实现）           |
| data_fetcher.py      | 合并到 data-fetcher 服务，作为新路由                               |
| engine/main.py       | 合并到 data-fetcher 服务，作为后台任务                             |

**权衡**：

- akshare 的 Python SDK 功能最全，Go 需要自行封装 HTTP 调用，但 REST API 接口稳定
- yfinance 的 Go 替代库（如 go-yfinance）功能不如 Python 版完整，但核心行情数据获取已满足需求

### 新架构

| 语言             | 服务           | 职责                                   | 框架/库                  |
| ---------------- | -------------- | -------------------------------------- | ------------------------ |
| TypeScript/React | frontend-api   | API 网关、路由调度、降级编排、前端托管 | Express 4 + React + Vite |
| Go               | engine-service | 回测计算、蒙特卡洛、组合优化、矩阵运算 | Gin + gonum + pgx        |
| Go               | data-service   | 标的搜索、价格查询、数据获取、baostock | Gin + baostock + pgx     |

**服务间通信**：HTTP JSON（保持不变），通过 K8s Service 互联。

**降级策略**：

- engine-service 不可用 → fail-closed 503 + Retry-After（ADR-031，不再降级到 Node 备用引擎）
- data-service 不可用 → PostgreSQL → Go 数据服务实时获取缺失 ticker（ADR-007，不再降级到 JSON 文件）

### 迁移路径

1. **Phase 1**：Go engine-service 开发（并行运行，与 Rust 引擎 A/B 测试）
2. **Phase 2**：Python 数据获取迁移到 data-service（合并到 Go 服务）
3. **Phase 3**：Rust 引擎下线，Python 运行时从 Docker 镜像移除
4. **Phase 4**：CI/CD 简化（移除 Rust/Python 构建步骤）

## Consequences（后果）

**正面后果**：

- **运维复杂度降低 50%**：从 4 个运行时（Node/Rust/Go/Python）减少到 2 个（Node/Go）
- **CI 构建时间减少 60%**：移除 Rust 编译（5-10 分钟）和 Python 依赖安装
- **团队门槛降低**：只需掌握 Go + TypeScript，招聘池扩大 3-5 倍
- **调试体验改善**：Go 的 OTel 集成成熟，跨服务追踪不再在 Rust 边界断裂
- **Docker 镜像精简**：移除 Python 运行时和 Rust 构建工具链，总镜像体积减少约 500MB
- **部署一致性**：Go 服务统一使用 Gin 框架，中间件和错误处理模式一致

**负面后果**：

- **计算性能下降**：Go 的数值计算性能约为 Rust 的 70-90%，蒙特卡洛模拟等 CPU 密集型任务可能慢 10-30%
- **Python 数据生态损失**：akshare/yfinance 的 Python SDK 功能最全，Go 需要额外封装
- **迁移工作量**：Rust 引擎约 2000 行代码需用 Go 重写，Python 脚本约 500 行需迁移
- **过渡期双运行**：Phase 1-2 期间 Rust 和 Go 引擎并行运行，增加临时复杂度

**风险缓解**：

- 性能下降：通过 Go 的 sync.Pool 对象复用和 gonum 的 BLAS 加速弥补
- 数据生态：优先使用 REST API 替代 Python SDK，必要时保留 Python 作为离线批处理工具
- 迁移风险：并行运行 + A/B 测试确保 Go 引擎结果与 Rust 引擎一致
