# ADR-021: 代码复杂度量化门控

> **企业理由**：代码质量不应只是主观感受。圈复杂度、认知复杂度、函数长度是可度量、可门控的工程指标。无量化门控时复杂度单调上升，最终形成无人敢改的"上帝函数"，使变更成本与缺陷率指数增长。

## Context

审计（维度4 / 4.1-4.2）发现：

- `eslint.config.js` 无任何复杂度规则（`complexity`/`max-lines`/`max-depth`）。
- 存在 13+ 个超 50 行函数，最长 `runSinglePortfolio` 269 行（`api/engine/portfolio.ts:157-425`），`executeOptimization` 223 行。
- `tsconfig.json` 关闭了 `noUnusedLocals/Parameters`，进一步弱化质量信号。

后果：复杂函数难以单元测试覆盖全分支（直接拉低关键文件 90% 覆盖率门槛达成难度），review 噪音大，缺陷集中。

## Decision

在 ESLint 启用内置复杂度规则（无需新增依赖），阈值采用企业基准：

| 规则                     | 阈值 | 依据                           |
| ------------------------ | ---- | ------------------------------ |
| `complexity`             | 15   | McCabe；>10 警惕，>15 重构红线 |
| `max-depth`              | 4    | 深嵌套 = 认知负荷激增          |
| `max-lines-per-function` | 80   | 单函数应聚焦单一职责           |
| `max-params`             | 5    | 过多参数提示应引入参数对象/VO  |
| `max-nested-callbacks`   | 3    | 回调地狱信号                   |

**渐进式落地**：先以 `warn` 暴露存量热点（与本仓既有 `no-unused-vars` 渐进策略一致），
逐项重构后升级为 `error`。配套工具：`npm run simplify`（jscpd 重复检测）、`npm run deadcode`（knip 死代码）。

## Consequences

- 优势：复杂度可见、可趋势化；新代码受硬约束；为后续提升关键文件覆盖率扫清障碍。
- 劣势：存量热点产生 warn 噪音——刻意为之，作为"技术债热点地图"驱动重构。
- 工程权衡：用 warn 而非 error 起步，避免一次性阻断 CI；但需设定升级为 error 的完成判定（DoD），防止 warn 永久沉默。
- 工具选型：内置规则零依赖；如需认知复杂度（Cognitive Complexity）可引入 `eslint-plugin-sonarjs`；商业方案 SonarQube/Codacy。
