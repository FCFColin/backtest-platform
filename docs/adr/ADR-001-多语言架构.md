# ADR-001: 多语言架构（Rust + Node + Go + Python）

> **企业理由**：单一语言无法同时满足计算密集型、I/O 密集型和数据生态三个维度的需求，多语言架构让每种语言在最擅长的领域发挥优势，避免"一刀切"带来的性能或生态妥协。

## Context（背景和驱动力）

回测平台需要同时满足以下三类需求：

1. **计算密集型**：组合回测、蒙特卡洛模拟、组合优化涉及大规模矩阵运算和并行计算，对 CPU 性能和内存安全要求极高。
2. **I/O 密集型 + API 网关**：前端 API 路由、请求调度、降级编排、熔断器管理需要成熟的 HTTP 框架和丰富的 npm 生态。
3. **金融数据生态**：A 股数据（baostock）和美股数据（akshare / yfinance）的 Python SDK 最为成熟，Go 和 Rust 生态中缺少等价替代。

单一语言方案的困境：

- **纯 Node.js**：矩阵运算性能差（V8 JIT 对数值计算优化不足），蒙特卡洛 10 万次模拟耗时是 Rust 的 10-100 倍。
- **纯 Rust**：缺乏成熟的 Web 框架生态（前端 SSR、i18n、状态管理），开发效率低；Python 金融库无法替代。
- **纯 Python**：并发能力弱，GIL 限制 CPU 并行，不适合作为 API 网关。
- **纯 Go**：数值计算库（矩阵、线性代数）远不如 Rust 的 nalgebra 成熟，Python 金融数据 SDK 无法使用。

## Decision（决策内容）

采用四语言微服务架构，各语言职责明确：

| 语言       | 服务     | 职责                                   | 框架/库                        |
| ---------- | -------- | -------------------------------------- | ------------------------------ |
| TypeScript | Node API | API 网关、路由调度、降级编排、前端托管 | Express 4 + tsx                |
| Rust       | 回测引擎 | 组合回测、蒙特卡洛、组合优化、矩阵运算 | actix-web 4 + nalgebra + rayon |
| Go         | 数据服务 | 标的搜索、价格查询、baostock 客户端    | gin + baostock SDK             |
| Python     | 数据获取 | akshare / yfinance 数据拉取            | subprocess 调用                |

服务间通信：HTTP JSON（同步调用），通过 Docker Compose 内部网络互联。

降级策略：Rust 不可用 → Node.js 备用引擎；Go 不可用 → Python 子进程或本地 JSON 文件。

## Consequences（后果）

### 正面

- **性能最优**：Rust 处理计算密集型任务，利用 rayon 并行 + nalgebra SIMD，回测速度比 Node.js 快 10-100 倍。
- **生态最优**：每种语言使用最成熟的库，避免"重新造轮子"（如用 Rust 重写 baostock 协议）。
- **独立部署**：各服务独立构建、部署、扩缩容，Rust 引擎可单独水平扩展。
- **故障隔离**：单语言服务崩溃不影响其他服务，熔断器防止雪崩。

### 负面

- **运维复杂度**：4 个运行时环境（Node / Rust / Go / Python），Docker 镜像构建和维护成本高。
- **团队门槛**：开发者需掌握多种语言，招聘和培训成本增加。
- **调试难度**：跨服务调用链追踪需要 request_id 关联，排障时需查看多个服务日志。
- **构建时间**：Rust release 编译耗时较长（LTO + opt-level=3），CI 流水线需针对性优化。
