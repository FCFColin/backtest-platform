# ADR-002: JSON 文件存储

> **企业理由**：MVP 阶段的核心目标是验证回测引擎的计算正确性和前端交互体验，数据存储不是核心差异化能力。JSON 文件存储将开发精力集中在业务逻辑上，避免过早引入数据库带来的运维和迁移成本。

## Context（背景和驱动力）

回测平台当前的数据存储需求：

1. **行情数据**：约 500+ 只标的的历史价格（OHLCV），每只标的 1 个 JSON 文件（如 `data/market/tickers/SPY.json`）。
2. **CPI 数据**：中美 CPI 月度数据，2 个 JSON 文件（`cn_cpi.json` / `us_cpi.json`）。
3. **标的列表**：`universe.json` 存储标的元信息。
4. **无写入并发**：数据由 Go 服务或 Python 脚本离线拉取，运行时只读。
5. **单实例部署**：Docker Compose 单机部署，无分布式需求。

考虑的替代方案：
- **SQLite**：单文件数据库，查询能力更强，但需要 ORM 或 raw SQL，增加开发复杂度。
- **PostgreSQL**：生产级关系数据库，但 MVP 阶段引入运维负担（备份、连接池、迁移脚本）。
- **MongoDB**：文档数据库，JSON 原生存储，但引入额外服务进程和运维成本。

## Decision（决策内容）

采用 JSON 文件存储，数据目录结构如下：

```
data/
  market/
    tickers/          # 每只标的 1 个 JSON 文件
      SPY.json
      VTI.json
      ...
    cpi/              # CPI 数据
      cn_cpi.json
      us_cpi.json
    state/            # 元信息
      universe.json
```

读写策略：
- **Go 数据服务**：启动时加载 universe.json 到内存，按需 lazy-load 标的价格文件，使用 `sync.RWMutex` 保护并发读取。
- **Node API**：通过 Go 服务获取数据，降级时直接 `fs.readFileSync` 读取本地 JSON 文件。
- **Docker 部署**：通过 `volumes: ./data:/app/data:ro` 只读挂载。

### 迁移触发条件

当以下任一条件满足时，应启动从 JSON 文件存储到数据库的迁移：

| 触发条件                       | 迁移目标       | 理由                                     |
| ------------------------------ | -------------- | ---------------------------------------- |
| 标的数量 > 5,000               | SQLite         | 文件系统 inode 压力，目录遍历变慢        |
| 需要按日期范围查询（非全量加载）| SQLite / PostgreSQL | JSON 需全量加载后内存过滤，浪费内存     |
| 多实例部署（水平扩展）          | PostgreSQL     | 文件系统无法跨实例共享                   |
| 需要写入并发（用户自定义组合）  | SQLite / PostgreSQL | 文件锁竞争，数据一致性风险              |
| 单文件 > 50MB                  | SQLite         | JSON 解析内存峰值约为文件大小的 3-5 倍  |

## Consequences（后果）

### 正面

- **零运维**：无需数据库服务进程、连接池、备份策略，`docker compose up` 即可运行。
- **开发效率**：数据文件可直接用文本编辑器查看和调试，无需数据库客户端。
- **可移植性**：JSON 文件可直接复制、版本控制、跨环境同步。
- **启动速度**：Go 服务启动时仅加载 universe.json（~1MB），标的价格按需加载。

### 负面

- **查询能力弱**：无法执行 SQL 查询（如"找出 2024 年涨幅前 10 的标的"），需全量加载后内存过滤。
- **内存占用**：热门标的的价格文件全量加载到内存，500 只标的 × 25 年日线 ≈ 200MB。
- **无事务**：数据更新非原子操作，写入中断可能导致文件损坏。
- **并发写入风险**：多进程同时写入同一文件可能导致数据丢失（当前通过只读挂载规避）。
- **扩展性上限**：标的数量超过 5,000 后，文件系统性能下降，需迁移到数据库。
