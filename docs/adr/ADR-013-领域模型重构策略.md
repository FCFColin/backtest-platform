# ADR-013: 领域模型重构策略（DDD落地路径）

> **企业理由**：贫血模型导致业务逻辑散落、规则重复、耦合严重，渐进式 DDD 重构可在控制风险的前提下统一业务规则、提升可维护性。

## Context

当前系统使用贫血模型，业务逻辑散落在路由和服务文件中，存在以下问题：

- 6个完整算法（蒙特卡洛、PCA、LETG等）嵌入路由文件，路由文件承担了控制器和业务逻辑双重职责
- `shouldRebalance`在3处重复实现，修改时容易遗漏导致行为不一致
- 缺乏领域边界，回测参数校验、策略执行、结果聚合等逻辑耦合在同一个服务层
- 新增策略需要修改多个文件，违反开闭原则

DDD（领域驱动设计）可解决上述问题，但全量重构风险高、周期长。需要确定渐进式落地路径。

候选方案：

- **渐进式DDD**：先Value Object → 再Aggregate → 最后Domain Event，分阶段落地
- **革命式重构**：一次性按DDD重新设计领域模型，全面替换现有架构
- **不重构**：维持现状，通过代码规范和Code Review控制技术债

## Decision

采用渐进式DDD落地策略，按Value Object → Aggregate → Domain Event的顺序分阶段重构。

落地路径：

1. **Phase 1 - Value Object**：提取回测参数（BacktestParams）、策略配置（StrategyConfig）、组合权重（PortfolioWeight）等值对象，封装校验逻辑和不变式
2. **Phase 2 - Aggregate**：定义回测聚合根（BacktestSession），统一shouldRebalance等业务规则，划定事务边界
3. **Phase 3 - Domain Event**：引入领域事件（BacktestCompleted、RebalanceTriggered），解耦副作用（通知、审计、缓存更新）

## Consequences

- (+) 一次性全量重构风险高，渐进式可逐步验证每个阶段的正确性，降低回滚成本
- (+) Value Object是最安全的起点——纯值校验无副作用，不涉及事务边界调整，可独立测试
- (+) Aggregate阶段统一业务规则，消除shouldRebalance等重复逻辑，保证行为一致性
- (+) Domain Event阶段解耦副作用，为Outbox模式（ADR-014）提供事件源
- (-) 渐进式意味着长期存在新旧架构并存，代码库中同时存在贫血模型和富领域模型，增加认知负担
- (-) 放弃革命式重构——风险不可控，回测平台是生产系统，长时间不可用不可接受
- (-) 放弃不重构——技术债持续累积，重复逻辑和耦合问题会随功能增长恶化

## 实施状态

### Phase 1 — Value Object（已完成）

- `domain/value-objects/ticker.ts`、`domain/value-objects/weight.ts` 已落地
- 由 Portfolio 聚合根的 `fromDTO()` 在构造时净化输入

### Phase 2 — Aggregate（部分完成）

- `domain/aggregates/portfolio.ts`：Portfolio 充血模型已落地，私有 constructor +
  静态 fromDTO 工厂 + 内置 `validateWeightSum` 不变量校验
- `domain/aggregates/run.ts`：Run 聚合根已落地（ADR-013 Phase 2 延伸），
  封装回测运行状态机 `queued → running → completed/failed/cancelled`，
  支持 `start() / complete(result) / fail(reason) / cancel()` 转换，
  内置终态不可变校验（终态再转换抛 `DomainValidationError`）
- `repositories/backtestRunRepo.ts` 新增 `save(run)` UPSERT 入口，
  domain 层 'queued' ↔ DB 'pending' 在 repo 层做映射，不破坏 schema

### Phase 3 — Domain Event（已落地）

- `domain/events/EventDispatcher.ts`：`DomainEventDispatcher` + `allSettled` 语义，并导出 `RUN_STARTED/COMPLETED/FAILED/CANCELLED` 事件类型常量
- 事件由聚合根状态转换时构造：
  - `Run.create()` 产生 `RunStarted`
  - `Run.complete(result)` 产生 `RunCompleted`
  - `Run.fail(reason)` 产生 `RunFailed`
  - `Run.cancel()` 产生 `RunCancelled`
- Handler：
  - `application/backtestCompletedHandler.ts`：订阅 `BacktestCompleted`，
    持久化回测摘要到 `backtest_runs`（同步路径）
  - `application/runCompletedHandler.ts`：订阅 `RunCompleted`，
    仅做观测副作用（日志），不重复持久化——Run 聚合根本身已由 worker 的
    `save()` 落库，handler 再写会造成双写
- `application/backtest-service.ts`：在引擎调用前通过 `Run.create()` 分发 `RunStarted`，
  原有 `BacktestCompleted` 双通道分发（dispatcher + outbox）保留不变
- `queues/worker.ts`：`persistRunIfTenant` 改造为通过 Run 聚合根驱动状态机
  （`create → start → complete → save → pullEvents → dispatch`），
  替代原有直接 `createRun({...status:'completed'})` 写法

### 兼容性说明

- DB schema 保持 `pending/running/completed/failed`（未做 migration）
- domain 层 `cancelled` 在 repo 层映射为 `failed`（`DOMAIN_TO_DB_STATUS`）
- 现有 `BacktestCompleted` 事件与新增的 `Run*` 事件并行存在，前者负责摘要落库，
  后者负责细粒度状态机——不互相替代
