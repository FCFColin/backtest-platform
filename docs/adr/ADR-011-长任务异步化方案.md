# ADR-011: 长任务异步化方案（BullMQ + Redis vs pg-boss vs 自建）

> **企业理由**：CPU 密集型长任务若同步执行会阻塞事件循环导致服务不可用，异步化是保障多用户并发体验和服务可用性的必要基础设施。

## Context
参数优化（1000组合）和网格搜索（200组合）同步执行阻塞Node.js事件循环30-100s，期间所有其他HTTP请求被挂起，用户体验极差。Node.js单线程模型下，CPU密集型任务必须异步化处理，否则整个服务不可用。

当前问题：
- 同步回测占用事件循环，健康检查超时导致K8s重启Pod
- 用户无法取消正在运行的任务
- 多用户同时提交优化任务时互相阻塞

候选方案：
- **BullMQ + Redis**：Node.js生态最成熟的任务队列，基于Redis实现
- **pg-boss**：基于PostgreSQL的任务队列，无额外依赖
- **自建**：基于PostgreSQL SKIP LOCKED的简易队列

## Decision
采用BullMQ + Redis作为长任务异步化方案。

## Consequences
- (+) Node.js生态最成熟的任务队列，文档完善，社区活跃，生产验证充分
- (+) 支持优先级、重试（指数退避）、延迟、定时（repeatable）、并发控制（rate limiter），覆盖所有任务调度场景
- (+) 自带Dashboard（Bull Board），便于开发调试和运维监控任务状态
- (+) Redis已在项目规划中（ADR-012 Redis选型），不引入新的基础设施依赖
- (+) 支持任务进度上报（job.progress），前端可实时展示优化进度
- (+) 支持任务取消（job.remove），用户可中止长时间运行的优化
- (-) 引入Redis依赖增加运维复杂度——需保证Redis高可用（Sentinel/Cluster）
- (-) 放弃pg-boss——无新依赖但性能不如Redis（PostgreSQL行锁竞争在高并发下明显），且功能不如BullMQ完善（缺少优先级、进度上报）
- (-) 放弃自建——维护成本高，需自行实现重试、死信队列、并发控制等基础能力
