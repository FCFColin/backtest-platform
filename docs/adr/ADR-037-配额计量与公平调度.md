# ADR-037: 按计划配额、用量计量与租户公平调度

> **企业理由**：分层变现与滥用防护要求把"按计划限制资源消耗"落到请求与任务路径上：每个计划定义月度计算次数、单次标的数、异步并发与速率上限；超限拦截并引导升级；用量需计量以供配额判定与计费对账；共享 worker 不能被单个租户占满而饿死他人。

## Decision（决策内容）

### 配额定义（`packages/backend/src/config/planLimits.ts`）

- 每计划：`backtestsPerMonth`、`maxTickers`、`asyncConcurrency`、`rateLimitPerMin`。`free/pro/enterprise` 三档，未知计划回落 `free`（最严格，fail-safe）。
- 静态配额表与类型保留在 `config/planLimits.ts`；查表函数 `getPlanLimits` 与计费周期工具 `currentPeriod` 见 `packages/backend/src/services/planLimitsService.ts`。

### 用量计量（迁移 `012_usage.sql` + `packages/backend/src/services/usageService.ts`）

- `usage_events`（明细，审计/BI）+ `usage_counters`（按 `org/period/metric` 月度聚合，配额权威）；两表启用 RLS。
- 双写 DB + Redis 月度计数器（快路径读，DB 兜底跨实例一致性），任一失败不阻断主流程。

### 配额中间件（`packages/backend/src/middleware/quota.ts`）

- 计算/入队前校验单次标的数（超 → 422）与月度用量（达上限 → 402），放行后计量一次。
- 无活跃租户（匿名本地开发）与 `platform_admin` 放行，保持零摩擦与运维豁免。
- 接到所有计算挂载（含 legacy）；元数据查询失败时 fail-open 放行，避免误伤合法请求。

### 速率限制按租户

- `computeRateLimitKey` 优先 `req.tenantId`（或 JWT `tenant_id`），使同组织成员共享配额，避免多用户绕过。

### 租户公平调度（`packages/backend/src/queues/worker.ts`）

- 每租户在途任务计数（Redis `inflight:<org>`），超过 `asyncConcurrency` 时 `DelayedError` 延迟重试，让出名额；处理结束释放。Redis 异常时跳过门控（计数失效优于任务卡死）。

## Consequences（后果）

### 正面

- 变现分层与滥用防护可执行；用量可计费对账；单租户无法饿死共享队列。
- 配额读走 Redis 快路径，DB 持久兜底。

### 负面

- 计量在放行后乐观计数，可能计入随后失败的计算（配额按"尝试"计，可接受）。
- 公平调度依赖 Redis 计数；其不可用时退化为仅 worker 全局并发上限。
